# 测试与日志规则

> 本文件定义测试和日志的必写场景与格式。

---

## 一、日志规则

### 1.1 必须写日志的场景

| 场景 | 必写内容 | 示例 |
|------|---------|------|
| 网络请求 | 请求参数 + 响应结果 + 错误 | `[LanBridge] connect: ws://192.168.1.5:9720` |
| 状态变更 | 关键状态转换点 | `[Room] state: waiting → playing` |
| 异步 catch | 错误信息 + 上下文 | `[LlmManager] call failed: timeout, model=deepseek` |
| 条件分支的 else | 走了哪个分支 | `[Bidding] no valid bid, fallback to 0` |
| 数据同步 | 同步内容摘要 | `[FullSync] items: 12, players: 3, round: 2` |

### 1.2 日志格式

```
[模块名] 动作: 详情
```

- ✅ `[LanBridge] connect: ws://192.168.1.5:9720`
- ✅ `[AI] bid: playerId=p1, amount=85000, confidence=0.72`
- ✅ `[Warehouse] spawn: 12 items, occupancy=65%`
- ❌ `connected`
- ❌ `error!!!`
- ❌ `data: [object Object]`

### 1.3 日志级别

- `console.log`：正常流程信息
- `console.warn`：可恢复的异常（如降级、回退）
- `console.error`：不可恢复的错误（如连接断开、数据损坏）

### 1.4 禁止的日志

- ❌ 调试完成后不清理的 `console.log('debug:', data)`
- ❌ 打印敏感信息（密码、token）
- ❌ 循环中的高频日志（每帧/每秒多次）

## 二、测试规则

### 2.1 判断标准：它会不会无声无息地坏掉？

- **会** → 必须写测试
- **不会**（坏了立刻看得见）→ 可以豁免

### 2.2 必须写测试的逻辑

| 类型 | 示例 | 原因 |
|------|------|------|
| 校验/白名单/权限 | 出价范围校验 | 边界条件容易漏 |
| 状态机/级联逻辑 | 游戏状态流转 | 状态转换容易出错 |
| 解析器/编解码 | LLM 响应解析 | 格式变化会静默失败 |
| 路由派发 | WebSocket 消息路由 | 新消息类型容易漏处理 |
| 计算/算法 | AI 出价算法 | 数值错误不易发现 |
| 数据转换 | 藏品价值计算 | 精度/溢出问题 |

### 2.3 可以豁免测试的逻辑

- 纯 UI 微调（颜色、间距、动画参数）
- 一次性脚本
- 纯改名/移动文件
- 简单的赋值/透传

### 2.4 测试方法

本项目暂无测试框架，采用以下方式：

1. **修复 bug 时**：先在浏览器控制台写一段能复现 bug 的代码，确认复现后再修
2. **新功能开发时**：先定义成功标准（输入 → 预期输出），实现后在控制台验证
3. **关键算法**：在文件底部写 `// TEST:` 注释，包含测试用例

```javascript
// TEST: computeBid({ playerId: 'p1', clueRate: 0.5, qualityRate: 0.8 })
//       → 应返回 50000~150000 范围内的值
// TEST: computeBid({ playerId: 'p1', clueRate: 0, qualityRate: 0 })
//       → 应返回 0 或极低值
```

### 2.5 验证清单

每次修改后，检查以下项目：

- [ ] 修改的函数在浏览器中能正常调用
- [ ] 边界条件（空值、极值）不会崩溃
- [ ] 异步操作有 catch 处理
- [ ] 控制台无新增报错
