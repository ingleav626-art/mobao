# 核心行为规则

## 一、思考先行

1. 不确定就问：如果需求有多种理解，列出所有可能的理解让用户选择，不要沉默地选一个
2. 不要假设：如果缺少关键信息（如参数类型、调用方、预期行为），先问再写代码
3. 发现更简方案时主动提出：如果你发现用户的方案可以更简单实现，说出来，不要默默执行复杂方案
4. 理解上下文再动手：修改代码前，先读取相关文件理解现有实现，不要凭猜测修改

## 二、简洁优先

1. 只写被要求的代码：不添加未请求的功能、配置、错误处理、抽象层
2. 3行相似代码优于1个过早抽象：不要为一次性操作创建工具函数
3. 200行能解决的不要写500行：如果发现代码膨胀，停下来简化
4. 不为不可能发生的场景写防御代码：只在系统边界（用户输入、外部API）做验证

## 三、精准修改

1. 只改必须改的：不要顺手"优化"相邻代码、注释、格式
2. 匹配现有风格：即使你不喜欢现有代码风格，也保持一致
3. 自己的改动产生的死代码必须删除：你删了一个函数，它不再被调用的 import 也要删
4. 已有的死代码不要擅自删除：发现时提醒用户，等用户确认后再删
5. 每一行改动都必须能追溯到用户的需求

## 四、目标驱动

1. 定义可验证的成功标准：
   - "修复 bug" → 先写能复现 bug 的测试，再修到测试通过
   - "添加功能" → 明确输入/输出示例，实现后验证
   - "重构" → 确保重构前后测试都通过
2. 多步任务先列计划：写出步骤和每步的验证条件，不要一口气改完再检查
3. 批量改完再统一检查：不要改一行就验证一次，攒成一波再查

## 五、旧代码清理

1. 禁止注释式删除：❌ `// const oldVar = ...` → 要删就真删，Git 有历史
2. 禁止抑制警告掩盖问题：❌ `eslint-disable`、`// @ts-ignore` → 要修就真修
3. 重构后旧接口必须清理：如果旧接口不再被任何地方调用，删除实现代码
4. 每次改动顺手清理：自己改动产生的无用 import/变量/函数 → 必须立即删除
5. 发现已有死代码：提醒用户，不擅自删除

## 六、提交纪律

1. 一个功能点 + 检查通过 = 立刻提交：不要攒一大坨
2. commit 信息说清"改了什么、为什么"：不要只写 "fix" 或 "update"
3. 只暂存自己改的文件：不要 `git add .`，逐个添加
